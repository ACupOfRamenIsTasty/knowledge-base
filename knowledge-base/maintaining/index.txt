10:Tfcd,<h1>Maintaining the Knowledge Base</h1>
<h2>How It Works</h2>
<ul>
<li>Users and Developers collaborate to keep content current</li>
<li>Users prepare markdown (<code>.md</code>) files — one file per lesson</li>
<li>Devs integrate those files into the app</li>
<li>Everyone sees the published knowledge</li>
</ul>
<h2>Best Practices</h2>
<ul>
<li>Use bullet points and step-by-step instructions</li>
<li>Connect content to everyday tasks where possible</li>
<li>Keep writing simple, concise, and adaptable to changing workflows</li>
</ul>
<p><strong>If you write for hours about a tool that soon becomes obsolete, you&#39;re not getting those hours back.</strong></p>
<h2>Adding Content via Markdown (Users)</h2>
<p><strong>Assume that users are not tech-savvy.</strong></p>
<h3>Write content using <a href="https://write-box.appspot.com/" target="_blank" rel="noopener noreferrer" style="color: #2563eb; text-decoration: underline;">Write-Box</a></h3>
<img src="/knowledge-base/writebox.png" style="margin: 16px 0; border-radius: 8px; border-width: 1px; border-color: black" />

<h3>Prompt the AI to convert your text to markdown format</h3>
<img src="/knowledge-base/copy-prompt.png" style="margin: 16px 0; border-radius: 8px; border-width: 1px; border-color: black" />

<p><strong>Copy paste this prompt along with your raw text:</strong></p>
<pre><code>Convert the following text into a markdown file. Follow these rules:

- Use one H1 heading (#) for the title at the top of the file. Use ## and ### for all subsequent headings
- Always use bullet points instead of paragraphs to keep content concise
- For any links, use the format [text](url)
- Keep everything brief and scannable — avoid unnecessary words
- Use the following as a format reference:

# Introduction

## What is the Knowledge Base?
- The single source of truth for all knowledge
- Helps manage scattered documents across teams

## Objective
- Have all staff update the app&#39;s content periodically and before leaving
- With each iteration, the app covers more material and remains up-to-date
</code></pre>

<h2>When to Use Markdown</h2>
<p>Use markdown when:</p>
<ul>
<li>The topic requires quick, basic knowledge for various team members</li>
<li>The workflow changes frequently and needs regular updates</li>
<li>The content should be accessible to all staff</li>
</ul>
<p>Otherwise:</p>
<ul>
<li>Don&#39;t bother with writebox and AI prompting</li>
<li>Prepare a Word document and give it to Devs</li>
</ul>
<h2>Maintaining the Repo (Devs)</h2>
<ul>
<li>See <code>README.md</code> for more info</li>
<li>Maintain the same folder structure across <code>content/</code> and <code>public/</code></li>
<li>Insert markdown files in <code>content/</code></li>
<li>Insert images in <code>public/</code></li>
<li>Merge to <code>dev</code> branch and wait for pipeline to finish to see changes</li>
</ul>
